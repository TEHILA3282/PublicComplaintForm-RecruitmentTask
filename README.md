# מערכת פניות הציבור

## תיאור הפרויקט

הפרויקט הינו מערכת Full Stack להגשת פניות ציבור, המורכבת מממשק משתמש ב-Angular ומשרת ב-ASP.NET Core Minimal API.

המערכת מאפשרת מילוי טופס פנייה רב-שלבי, ביצוע ולידציות, העלאת קבצים, שליחת הנתונים לשרת וטיפול בצד השרת.

במסגרת המשימה נוספו:

- שלב "פרטי הפנייה" בהתאם לסקיצה.
- Endpoint חדש המחזיר דוח פניות חודשי בפורמט JSON.

# שיטת הבנייה

הפרויקט בנוי בשכבות, כאשר לכל שכבה אחריות ברורה.

- רכיבי Angular אחראים על התצוגה.
- Services אחראים על הלוגיקה המשותפת ותקשורת עם השרת.
- ה-API אחראי על הלוגיקה העסקית.
- השירותים בשרת מבצעים את העבודה בפועל.
- המודלים מגדירים את מבנה הנתונים המועברים בין הלקוח לשרת.

במשימה החדשה הקפדתי להשתלב במבנה הקיים של הפרויקט ולא להוסיף ארכיטקטורה חדשה, כדי לשמור על אחידות וקלות תחזוקה.

# דוח פניות חודשי

נוסף Endpoint חדש:

```
GET /reports/monthly-inquiries
```

ה-Endpoint מחזיר:

- מספר הפניות בחודש המבוקש.
- מספר הפניות בחודש הקודם.
- מספר הפניות באותו חודש בשנה שעברה.
- ההפרש לעומת החודש הקודם.
- ההפרש לעומת השנה הקודמת.

לפי דרישות המשימה נעשה שימוש בנתוני דמה, אך המבנה מאפשר בעתיד החלפה פשוטה לשאילתת SQL ללא שינוי ב-API.

## טיפול בשגיאות

בפרויקט נעשה שימוש בשתי רמות של טיפול בשגיאות:

### טיפול מרכזי בחריגות

חריגות שלא טופלו בתוך ה־Endpoint נתפסות באמצעות מנגנון מרכזי של ASP.NET Core, המבוסס על `UseExceptionHandler`.

הטיפול המרכזי מאפשר:

- החזרת תשובות שגיאה אחידות ללקוח.
- רישום החריגה ללוג במקום אחד.
- מניעת שכפול של `try/catch` בכל Endpoint.
- שמירה על קוד קצר וקריא יותר.
- מניעת חשיפה של מידע טכני פנימי בסביבת Production.

### טיפול מקומי באמצעות try/catch

במקומות שבהם נדרש טיפול ייחודי בפעולה מסוימת, נעשה שימוש ב־`try/catch` מקומי.

לדוגמה, ב־Endpoint של שליחת דוא"ל קיים `try/catch` סביב ההתקשרות עם שרת ה־SMTP. פעולה זו תלויה במשאב חיצוני ועלולה להיכשל בשל בעיית תקשורת, הרשאות, הגדרות שרת או Timeout. הטיפול המקומי מאפשר להחזיר תשובה מותאמת לכשל בשליחת הדוא"ל.

בנוסף, קיים טיפול מקומי בעת פענוח טקסט מסוג Base64. במקרה שהקלט אינו תקין, המערכת אינה מפסיקה את הבקשה כולה, אלא משתמשת בטקסט חלופי וממשיכה בתהליך.

העיקרון שנבחר הוא:

- להשתמש בטיפול המרכזי עבור חריגות כלליות ולא צפויות.
- להשתמש ב־`try/catch` מקומי רק כאשר ניתן לבצע טיפול ייעודי, להחזיר תשובה מותאמת או להמשיך את הפעולה בצורה בטוחה.
- לתעד את השגיאה בלוג ולא לחשוף ללקוח פרטים פנימיים כגון Stack Trace, סיסמאות או פרטי חיבור.

# תקשורת בין המערכות

התקשורת בין Angular לשרת מתבצעת באמצעות HTTP.

השרת מחזיר נתונים בפורמט JSON.

הפרויקט משתמש ב-Minimal API ולכן כל Endpoint נשאר קצר, בעוד שהלוגיקה העסקית נמצאת בשירותים ייעודיים.

---

# Cross Origin (CORS)

כאשר הלקוח והשרת רצים על כתובות שונות יש להגדיר מדיניות CORS.

בסביבת Production מומלץ לאפשר גישה רק לכתובות המורשות ולא להשתמש ב-AllowAnyOrigin.

# הוראות התקנה והפעלה

## דרישות מקדימות

יש לוודא שמותקנים במחשב:

- .NET 8 SDK
- Node.js
- npm

---

## הפעלת ה־Backend

1. פתחו Terminal בתיקיית:

```
PublicComplaintForm_API
```

2. הריצו:

```bash
dotnet restore
dotnet run
```

השרת יעלה בכתובת המופיעה ב־Terminal (לדוגמה `https://localhost:xxxx`).

---

## הפעלת ה־Frontend

1. פתחו Terminal נוסף בתיקיית:

```
PublicComplaintForm
```

2. הריצו:

```bash
npm install
npm start
```

לאחר ההפעלה ניתן לפתוח את המערכת בדפדפן בכתובת:

```
http://localhost:4200
```
