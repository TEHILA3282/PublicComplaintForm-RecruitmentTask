# מערכת פניות ציבור

## תיאור הפרויקט

הפרויקט הינו מערכת Full Stack להגשת פניות ציבור, המורכבת מממשק משתמש ב־Angular ומשרת ב־ASP.NET Core Minimal API.

המערכת מאפשרת למשתמש למלא טופס פנייה רב־שלבי, לבצע ולידציות, להעלות קבצים, לשלוח את הנתונים לשרת ולקבל תשובות בפורמט JSON.

במסגרת המשימה בוצעו השינויים הבאים:

- הוספת שלב "פרטי הפנייה" בהתאם לסקיצה שסופקה.
- שליחת נתוני הטופס לשרת.
- הוספת Endpoint חדש המחזיר דוח פניות חודשי בפורמט JSON.
- שמירה על הארכיטקטורה הקיימת של הפרויקט ללא שינוי מבנה המערכת.

---

# ארכיטקטורת המערכת

```
Angular
     │
HTTP / JSON
     │
ASP.NET Core Minimal API
     │
Business Services
     │
Dapper
     │
SQL Server
```

המערכת בנויה בארכיטקטורת Client-Server כאשר צד הלקוח וצד השרת מופרדים לחלוטין ומתקשרים באמצעות HTTP בפורמט JSON.

## שיטת הבנייה

הפרויקט בנוי בהפרדת אחריות בין שכבות.

## Frontend (Angular)

- אחריות על התצוגה באמצעות **Components**.
- אחריות על הלוגיקה המשותפת וקריאות ה־HTTP באמצעות **Services**.
- ניהול נתוני הטפסים והוולידציות באמצעות **Reactive Forms**.
- שימוש ב־**HttpClient** לצורך תקשורת מול השרת.
- שימוש ב־**Angular Material** ליצירת ממשק משתמש אחיד ועקבי.

## Backend (ASP.NET Core)

- ה־**Minimal API** אחראי על חשיפת ה־Endpoints.
- הלוגיקה העסקית ממומשת בשירותים ייעודיים.
- הגישה למסד הנתונים מתבצעת באמצעות **Dapper**.
- המודלים מגדירים את מבנה הנתונים המועברים בין הלקוח לשרת.

השירותים מוזרקים באמצעות **Dependency Injection** של ASP.NET Core, המאפשר הפרדת אחריות, תחזוקה נוחה, הרחבה עתידית ושיפור יכולת הבדיקות.

במסגרת המשימה הקפדתי להשתלב במבנה הקיים של הפרויקט, ללא שינוי הארכיטקטורה, כדי לשמור על אחידות הקוד וקלות התחזוקה..

# דוח פניות חודשי

נוסף Endpoint חדש:

```
GET /reports/monthly-inquiries
```

ה־Endpoint מחזיר:

- מספר הפניות בחודש המבוקש
- מספר הפניות בחודש הקודם
- מספר הפניות באותו חודש בשנה שעברה
- ההפרש מהחודש הקודם
- ההפרש מהשנה הקודמת

במסגרת המשימה נעשה שימוש בנתוני דמה, אך מבנה השירות מאפשר החלפה פשוטה לשאילתת SQL אמיתית ללא שינוי ב־API.

# טיפול בשגיאות

במערכת קיימות שתי רמות של טיפול בשגיאות.

## טיפול מרכזי

חריגות שלא טופלו ב־Endpoint נתפסות באמצעות מנגנון:

```
UseExceptionHandler()
```

הטיפול המרכזי מאפשר:

- תשובות שגיאה אחידות
- רישום לוגים במקום מרכזי
- מניעת שכפול try/catch
- קוד נקי וקריא
- מניעת חשיפת Stack Trace או מידע פנימי בסביבת Production

---

## טיפול מקומי

במקומות שבהם ניתן לטפל בכשל בצורה ייעודית נעשה שימוש ב־try/catch מקומי.

לדוגמה:

- שליחת דוא"ל באמצעות SMTP
- פענוח Base64
- פעולות מול משאבים חיצוניים

במקרים אלו ניתן להחזיר הודעת שגיאה מותאמת ואף להמשיך את ביצוע הפעולה כאשר הדבר אפשרי.

# Logging

המערכת משתמשת ב־log4net לצורך רישום:

- שגיאות
- אזהרות
- מידע על פעולות מערכת

הלוגים מאפשרים ניטור, תחקור תקלות ואבחון בעיות בסביבת Production.

# Memory Cache

נעשה שימוש ב־Memory Cache עבור נתונים הנקראים לעיתים קרובות ואינם משתנים בתדירות גבוהה, ובכך מצטמצמות פניות מיותרות למסד הנתונים ומשתפרים ביצועי המערכת.

---

# Dapper

הגישה למסד הנתונים מבוצעת באמצעות Dapper.

Dapper נבחר בזכות:

- ביצועים גבוהים
- שליטה מלאה על שאילתות SQL
- מיפוי פשוט ומהיר בין תוצאות השאילתה לאובייקטים

---

# תקשורת בין המערכות

Angular מתקשר עם השרת באמצעות HTTP.

כל המידע מועבר בפורמט JSON.

Minimal API אחראי על ה־Endpoints בלבד, בעוד שהלוגיקה העסקית נמצאת בשירותים ייעודיים.

---

# Cross Origin (CORS)

כאשר הלקוח והשרת רצים על כתובות שונות מוגדרת מדיניות CORS.

בסביבת Production מומלץ:

- לאפשר גישה רק לכתובות המורשות.
- להימנע משימוש ב־AllowAnyOrigin.
- להגדיר Origins מורשים בלבד.

```bash
dotnet run
```

```
https://localhost:5001
```

```
PublicComplaintForm
```

```bash

```

או

```bash
ng serve
```
